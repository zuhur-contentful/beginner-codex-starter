Start here — Day 1 checklist

Before reading anything else, get the repo set up. This takes about 15 minutes.

  1. Go to github.com/zuhur-contentful/beginner-codex-starter
  2. Click Use this template → Create a new repository → name it after what you want to build
  3. Clone it to your computer: open Terminal, run git clone https://github.com/YOUR-USERNAME/YOUR-REPO.git
  4. Open the folder in the Codex App (download from github.com/openai/codex/releases)
  5. Edit CLAUDE.md — replace PROJECT_DESCRIPTION with one sentence about what you're building
  6. In Terminal, inside your project folder: cp .env.example .env — then open .env and generate a SECRET_KEY with the command shown in the file
  7. Create a virtual environment and install packages: 
    python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
  8. Run the tests to confirm everything works: python -m pytest tests/ -v — you should see 1 passed
  9. In the Codex App, type: "I want to build [describe your idea]. Help me plan the first feature."
Stuck on any of these steps? Open the ChatGPT desktop app and say: "I'm a complete beginner setting up a Python project on Mac. I'm on step [N] of a setup checklist and it isn't working. Here's what happened: [paste error]." It will walk you through it.

Once this is done, read the rest of the guide in order. Each stage tells you what to learn, what to build, and which Codex skills to use.

What you're going to build

By the end of this guide you will have built a real web app — something that runs in a browser, saves data, and uses AI. Something you want to use, not a tutorial project.

Examples of things beginners have built:

  • A private journal that summarises your week using GPT
  • A habit tracker that sends you an encouraging message each morning
  • A reading list with AI-generated summaries of each book
  • A recipe organiser that suggests meals based on what's in the fridge
  • A tool that emails a daily digest of things you care about

None of these require a computer science degree. All of them are buildable by someone who has never coded before.

The new rule with AI You can now ask AI to set up your machine, plan your project, write the first version of every feature, explain every error, and review your code. Your job is to understand what it's doing — not to memorise syntax.

Your tools

OpenAI makes several products. They're different and the naming is confusing. Here's what you need to know:

ChatGPT desktop app
Chat interface. Lives in your dock. Cannot edit your files.
→ Use for: brainstorming, questions, planning, learning
Codex App ★ Start here
GUI coding agent. Reads your files, writes code, runs commands, has Git built in.
→ Use for: actually building things
Codex CLI
Same as the Codex App but in your terminal. Keyboard-driven.
→ Use if: you prefer terminal over GUI
VS Code + ChatGPT extension
Chat panel inside your editor. Sees the file you're editing.
→ Use for: inline help without switching windows
ChatGPT is not deterministic The same question can give a different answer each time. Great for brainstorming and learning. Never paste code you don't understand into your project — you won't be able to fix it when it breaks.

You need a ChatGPT Plus subscription ($20/month) for all of these. The Codex App uses OAuth login — no extra API key needed to build with it.

Your starter repo — what's in it and how to use it

The template at github.com/zuhur-contentful/beginner-codex-starter is a ready-to-go project folder. Everything is pre-configured so you can open it in the Codex App on Day 1 and start building without fighting with setup.

How to start Go to the repo → click Use this template → Create a new repository → name it after what you're building → clone it to your computer → open the folder in the Codex App.

What's in the folder

File / folderWhat it isWhen you touch it
CLAUDE.md Project context file. Codex reads this automatically every session — it tells Codex what your stack is, where your files are, and your security rules. Day 1: fill in PROJECT_DESCRIPTION, your repo URL, and your deploy URL. Update it as your project grows.
.env.example A template listing every secret your app needs. Safe to commit — it has no real values. Day 1: run cp .env.example .env and fill in your real values. Never commit the actual .env.
.env (you create it) Your real secrets — API keys, database URLs. Already in .gitignore so it can never be accidentally pushed. Whenever you add a new secret: add the key to .env.example (blank value) and the real value to .env.
requirements.txt The list of Python packages your app depends on. Anyone who clones your repo runs pip install -r requirements.txt to get the same environment. After every pip install <package>: run pip freeze > requirements.txt to save it.
tests/ Automated tests. Contains a placeholder test so CI passes from Day 1. Add your own as you build. Stage 3: ask Codex to write tests for each feature before you deploy.
.github/workflows/ci.yml GitHub Actions CI — automatically runs your tests on every push. Already configured and working. You don't need to touch this to start. It just works. Read the GitHub Actions Quickstart in Stage 3 when you want to understand it.
.githooks/pre-commit A script that checks for accidentally included secrets before every commit. Blocks the commit if it finds an API key pattern. You don't need to configure this. It runs automatically when you use the commit skill.
.mcp.json Configures MCP servers — tools that extend what Codex can do. Pre-configured with the GitHub MCP so Codex can see your issues and PRs. Add your GITHUB_PERSONAL_ACCESS_TOKEN to .env to activate it.
.codex/skills/ Your five pre-built skills: plan.md, review.md, debug.md, commit.md, learn.md. Codex reads these automatically and follows their instructions when you say the trigger phrase. Use them from Day 1. You can edit any skill file to change how Codex behaves.
guide.html This document. Open in any browser. File → Print to save as PDF. Reference whenever you need it. Keep it in your project folder.
SKILLS.md Plain-text version of the skills guide — explains each skill and when to use it. Quick reference while in the Codex App.

Day 1 checklist

  1. Click Use this template on GitHub → create your own repo
  2. Clone it: git clone https://github.com/YOUR-USERNAME/YOUR-REPO.git
  3. Open the folder in the Codex App
  4. Edit CLAUDE.md — fill in PROJECT_DESCRIPTION and your repo URL
  5. Run cp .env.example .env — fill in SECRET_KEY (generate it with the command in the file)
  6. Create a virtual environment: python3 -m venv venv && source venv/bin/activate
  7. Install packages: pip install -r requirements.txt
  8. Run the placeholder test to confirm everything works: python -m pytest tests/ -v
  9. In the Codex App, type: "I want to build [describe your idea]. Help me plan the first feature."
CLAUDE.md is how Codex knows your project Every time you open the Codex App, it reads CLAUDE.md first. This is why it doesn't ask you "what stack are you using?" every session — it already knows. Keep it updated as your project grows: add new routes, new dependencies, anything important. The more context it has, the better its suggestions will be.

Your pre-built Codex skills

This repo includes five skills in .codex/skills/. Say the trigger phrase in the Codex App and it will follow the instructions automatically.

Say thisSkillWhat it doesUse it when
plan [feature] Plan Asks clarifying questions, scopes the smallest useful version, saves a plan file — no code until you confirm Before every new feature
review my code Review Reads your git diff, flags security issues and bugs, explains every problem in plain English with the fix shown After writing any code
debug this Debug One fix at a time, explains what caused the bug and why the fix works When something is broken
commit this Commit Scans for secrets, shows the diff, writes a proper commit message, asks for confirmation When saving work
explain this Learn Three levels: one sentence → line by line → why it's written that way When you don't understand code
The recommended workflow plan → build → explain this → review my code → commit this → git push. Follow this loop for every feature and you'll rarely get stuck without knowing why.

When things break

Every developer follows the same debugging loop. Get comfortable with it.

  1. Read the full error message — the last line is usually the actual problem
  2. Google the exact error message in quotes
  3. Say "debug this" in the Codex App — paste the full error and describe what you were doing
  4. Make one change at a time
  5. Run again and see what changed
  6. Repeat
